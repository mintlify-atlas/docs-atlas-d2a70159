## Overview

DAOs (Data Access Objects) define the interface between the application and the Room database. All DAOs are located in `data/dao/` and use Room annotations to generate SQL queries at compile time.

### Upsert by Code

Insert or update products based on their unique code (SKU).

```kotlin
@Insert(onConflict = OnConflictStrategy.IGNORE)
suspend fun insertIgnore(product: Product): Long

@Query("""
    UPDATE producto
    SET name = :name, price = :price, stock = :stock, descripcion = :desc
    WHERE code = :code
""")
suspend fun updateByCode(name: String, price: Double, stock: Int, desc: String, code: String): Int

@Transaction
suspend fun upsertByCode(p: Product) {
    val id = insertIgnore(p)
    if (id == -1L) {
        updateByCode(p.name, p.price, p.stock, p.descripcion, p.code)
    }
}
```

**Location:** `data/dao/ProductDao.kt:42`

The `upsertByCode` transaction attempts insertion first. If a product with that code already exists (returns -1), it updates the existing record instead.

### Upsert by Name

```kotlin
@Insert(onConflict = OnConflictStrategy.IGNORE)
suspend fun insertIgnore(c: Category): Long

@Query("SELECT * FROM categoria WHERE nombre = :nombre LIMIT 1")
suspend fun findByNombre(nombre: String): Category?

@Transaction
suspend fun upsertByName(nombre: String): Long {
    val trimmed = nombre.trim()
    val insertedId = insertIgnore(Category(nombre = trimmed))
    if (insertedId != -1L) return insertedId
    // Already exists: return its real id
    return findByNombre(trimmed)?.id ?: -1L
}
```

**Location:** `data/dao/CategoryDao.kt:35`

Attempts to insert a category. If it already exists (unique constraint on name), returns the existing category's ID instead.

## Room Annotations Guide

### @Query

Defines custom SQL queries. Room validates syntax at compile time.

```kotlin
@Query("SELECT * FROM producto WHERE code = :code")
suspend fun findByCode(code: String): Product?
```

### @Insert

Generates INSERT statements. Use `onConflict` to handle constraint violations.

```kotlin
@Insert(onConflict = OnConflictStrategy.IGNORE)
suspend fun insertIgnore(product: Product): Long
```

Conflict strategies:
- `IGNORE`: Skip insert if constraint violated
- `REPLACE`: Delete old row and insert new one
- `ABORT`: Default - throw exception

### @Update

Updates existing entities based on primary key.

```kotlin
@Update
suspend fun updateProduct(product: Product)
```

### @Delete

Deletes entities based on primary key.

```kotlin
@Delete
suspend fun deleteProduct(product: Product)
```

### @Transaction

Ensures operations run atomically. Required for multi-step operations and relationship queries.

```kotlin
@Transaction
suspend fun upsertByCode(p: Product) {
    val id = insertIgnore(p)
    if (id == -1L) {
        updateByCode(p.name, p.price, p.stock, p.descripcion, p.code)
    }
}
```

## Best Practices

<Card title="Use Flow for Reactive Queries" icon="wave">
  Flow automatically emits updates when database changes. Perfect for UI observation.
</Card>

<Card title="Suspend Functions for One-Shot Operations" icon="bolt">
  Use `suspend fun` for insert, update, delete operations. Automatically runs on background thread.
</Card>

<Card title="Batch Operations" icon="boxes">
  Insert multiple items with `insertAll(list)` instead of looping - much faster.
</Card>

<Card title="Transaction Boundaries" icon="shield-check">
  Mark methods with `@Transaction` when consistency matters (e.g., sale + details).
</Card>

<Card title="COALESCE for Aggregates" icon="calculator">
  Use `COALESCE(SUM(total), 0.0)` to return 0 instead of NULL when no rows match.
</Card>

<Card title="Local Timezone Queries" icon="clock">
  Use `'localtime'` in SQLite date functions for accurate day boundaries.
</Card>
